O que o projeto faz

Aplicativo desktop para gerenciar doações de itens (roupas, livros, alimentos, etc.) com cadastro, login, catálogo visual e upload de imagens.

Autenticação

Cadastro, login, edição de perfil. Sessão única ativa via Singleton.

Catálogo

Grade de cards com busca, ilustração por categoria, edição inline e exclusão com confirmação.

Upload de imagens

FileChooser nativo, cópia para diretório do usuário, preview e renderização real no card.

Stack técnica

  • Java 17 — records, switch moderno, text blocks (SQL multiline).
  • JavaFX 21 — UI declarativa via FXML + CSS estilo web.
  • SQLite (sqlite-jdbc) — banco embutido em arquivo único (doacao.db).
  • Maven — build, dependências, mvn javafx:run.
  • JUnit 5 — testes unitários.
  • JPMS (module-info) — modularização explícita.

POO — 4 pilares (lembrete)

1. Encapsulamento

Campos private + getters/setters. Estado só muda por métodos da classe.

No projeto: Item, Usuario.

2. Abstração

Esconder detalhes. Quem usa ItemService.cadastrar() não sabe que existe SQL embaixo.

3. Herança

Reuso por extensão. App extends Application herda ciclo de vida do JavaFX.

4. Polimorfismo

Mesma interface, comportamento diferente. equals() sobrescrito em Usuario.

Encapsulamento na prática

public class Item {
    private int id;            // estado interno protegido
    private String nome;
    private String imagemPath;

    public String getNome()        { return nome; }
    public void   setNome(String n) { this.nome = n; }
}

Acesso direto a item.nome é proibido. Quem precisa muda via setter — facilita validar, logar, ou trocar a representação interna depois sem quebrar quem usa.

Abstração — esconder o "como"

// Controller só pede ao Service. Não sabe que existe DAO, SQL, conexão.
boolean ok = itemService.cadastrar(nome, categoria, descricao, qtd, path);

Trocar SQLite por arquivo JSON, REST, ou MongoDB = mexer só no DAO. Controller fica intacto. Isso é abstração funcionando.

Regra de ouro: "o chamador não deveria precisar saber detalhes técnicos para fazer seu trabalho".

Herança — reuso por extensão

public class App extends Application {
    @Override
    public void start(Stage stage) throws Exception {
        // JavaFX chama isso pra mim. Eu só implemento.
    }
}

Application é uma classe abstrata da biblioteca. App herda ciclo de vida (start, stop, launch). Reuso real — sem reescrever janela, threads, evento.

Cuidado: herança gera acoplamento. Use só quando há relação verdadeira "é-um".

Polimorfismo — comportamento por contrato

@Override
public boolean equals(Object o) {
    if (this == o) return true;
    if (!(o instanceof Usuario)) return false;
    Usuario outro = (Usuario) o;
    return Objects.equals(nomeUsuario, outro.nomeUsuario);
}

List.contains(usuario) chama equals. O Java não sabe a regra de igualdade de Usuario — você ensina sobrescrevendo. Mesmo método, comportamento diferente por classe = polimorfismo.

Singleton — Sessao

public final class Sessao {
    private static Sessao instancia;
    private Usuario usuarioLogado;

    private Sessao() {}            // construtor privado!

    public static Sessao getInstancia() {
        if (instancia == null) instancia = new Sessao();
        return instancia;
    }
}

Quando usar: exatamente uma instância faz sentido (sessão, config global, cache).

Cuidado: Singleton é estado global — dificulta testes. Use com parcimônia.

DAO — Data Access Object

public class ItemDAO {
    public boolean salvar(Item item) {
        String sql = "INSERT INTO itens ... VALUES (?, ?, ?, ?, ?)";
        try (Connection c = Database.conectar();
             PreparedStatement ps = c.prepareStatement(sql)) {
            ps.setString(1, item.getNome());
            // ...
            ps.executeUpdate();
            return true;
        } catch (SQLException e) { return false; }
    }
}

Toda interação com banco passa por aqui. SQL nunca vaza para Controller. Banco trocou? Reescreva o DAO, o resto não percebe.

Service Layer — regras de negócio

public boolean cadastrar(String nome, String categoria,
                         String desc, int qtd, String imagem) {
    if (nome == null || nome.isBlank()) return false;
    if (categoria == null || categoria.isBlank()) return false;
    if (qtd <= 0) return false;

    Item item = new Item(nome.trim(), categoria.trim(), desc, qtd, imagem);
    return itemDAO.salvar(item);
}

Service valida e orquestra. DAO só persiste. Controller só decide quem chamar. Cada camada tem uma responsabilidade (SRP do SOLID).

Decisões de arquitetura

SQLite (não MySQL)

Zero configuração, arquivo único, ótimo para projeto acadêmico desktop.

FXML separado do código

Designer mexe na UI sem tocar em Java. Recompilação só do FXML.

Service entre Controller e DAO

Validação fica em um lugar só. Não duplicar em cada controller.

Imagens fora do banco

Arquivo copiado para ~/.projeto-doacao/imagens/. Só o caminho vai pro DB. Mais leve.

Migração via PRAGMA

ItemDAO verifica colunas e faz ALTER TABLE sem quebrar DB antigo.

Pacotes por feature

Tudo de "auth" junto. Coesão alta, acoplamento baixo.

SOLID em mini-doses

  • Single Responsibility — DAO só persiste, Service só valida.
  • Open/Closed — adicionar nova categoria não exige mexer no Catálogo (mapa de emoji aberto a extensão).
  • Liskov — App pode substituir Application em qualquer ponto que JavaFX chame.
  • Interface Segregation — controllers expõem só os @FXML que o FXML precisa.
  • Dependency Inversion — Controller depende da abstração Service, não do DAO direto.

Boas práticas aplicadas

  • PreparedStatement em todo SQL — previne SQL Injection.
  • try-with-resources — fecha conexão/statement sempre.
  • Construtor privado em classes utilitárias (Navegacao, Database, ImagemStorage) — impede instanciar.
  • final class em utilitários — impede herança acidental.
  • module-info.java — só pacotes que precisam ficam opens para JavaFX.
  • UUID em nomes de arquivos de imagem — evita colisão.

O que poderia evoluir

  • Hash de senha (BCrypt) — hoje senha está em claro no DB.
  • Interface Repository entre Service e DAO — facilita mock em testes.
  • Observer/Listener no ItemService para auto-refresh do catálogo.
  • Internacionalização via ResourceBundle.
  • Builder para Item (já tem 4 construtores).
  • Testes de integração com banco em memória.
